Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau's algorithm is a recursive method for evaluating a point on a Bezier curve at a specific parameter \( t \in [0, 1] \). Given \( n \) control points, the algorithm repeatedly applies linear interpolation between consecutive pairs of points, producing \( n-1 \) intermediate points at each level. This process continues until a single point remains, and that point lies exactly on the Bezier curve at parameter \( t \).

The interpolation at each step is: \[ p'_i = (1 - t) \cdot p_i + t \cdot p_{i+1} \] In my implementation, evaluateStep takes the current vector of points and performs exactly one level of this reduction. It iterates over consecutive pairs, computes the lerp using the member variable t, and returns the resulting vector of \( n-1 \) points. To fully evaluate the curve at a given \( t \), this function is called repeatedly until only one point remains.

Part 2: Bezier surfaces with separable 1D de Casteljau

De Casteljau's algorithm extends naturally to Bezier surfaces using a separable approach. A Bezier surface is defined by an \( n \times n \) grid of control points and two parameters \( u \) and \( v \). The key insight is that we can apply the 1D algorithm twice: first along one axis, then along the other.

Concretely, each row of \( n \) control points defines a Bezier curve parameterized by \( u \). We run the full 1D de Casteljau algorithm on each row at parameter \( u \), reducing each row down to a single point. This gives us \( n \) points — one per row — which themselves define a new Bezier curve parameterized by \( v \). We then run the algorithm once more on these \( n \) points at parameter \( v \), yielding the final single point that lies on the surface at \( (u, v) \).

In my implementation, evaluateStep performs one level of reduction on a vector of 3D points at a given \( t \), mirroring Part 1 but in 3D. evaluate1D repeatedly calls evaluateStep until only one point remains. Finally, evaluate loops over each row of controlPoints, calling evaluate1D at parameter \( u \) to get one point per row, then calls evaluate1D once more on those points at parameter \( v \) to get the final surface point.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

To compute the area-weighted vertex normal, we traverse all faces incident to the vertex using the half-edge data structure. Starting from the vertex's half-edge, we follow the twin()->next() pattern to loop around all neighboring faces. For each face, we retrieve two neighboring vertex positions and compute the cross product of the two edge vectors from the current vertex — this gives a vector whose direction is the face normal and whose magnitude is proportional to twice the face area. We accumulate these weighted normals and finally return the normalized sum. This naturally weights each face's contribution by its area without needing to explicitly compute area as a separate step.

Part 4: Edge flip

We implemented edge flip by carefully reassigning all half-edge pointers for the two affected triangles. Before writing any code, we drew out the two triangles \( (a, b, c) \) and \( (c, b, d) \) and labeled every half-edge, vertex, edge, and face. Then we drew the post-flip configuration \( (a, d, c) \) and \( (a, b, d) \) and determined where every pointer needed to go.

We used setNeighbors to reassign all six inner half-edges at once, then updated the halfedge() pointers for all four vertices, all five edges, and both faces. We also added early returns for boundary edges and the degenerate case where the two opposite vertices are the same, to avoid flipping edges that would create invalid geometry. A key debugging insight: it is much safer to reassign all pointers for all elements even if they haven't changed, rather than trying to figure out the minimal set of changes — this prevents hard-to-track bugs from missed pointer updates.

Edge split is more involved than edge flip because it introduces new mesh elements: 1 new vertex, 3 new edges, 6 new half-edges, and 2 new faces. Part 6: Loop subdivision for mesh upsampling

We implemented Loop subdivision in four steps following the recommended order. In Step A, for each old vertex we computed its updated position using \( (1 - n \cdot u) \cdot \text{position} + u \cdot \text{neighbor\_sum} \), where \( u = 3/16 \) if degree \( n = 3 \), else \( u = 3/(8n) \). For each edge, we computed the new midpoint vertex position using \( \frac{3}{8}(A + B) + \frac{1}{8}(C + D) \). All results are stored in newPosition without modifying actual positions yet. In Step B, we split every original edge (iterating only over the first nEdges() edges to avoid processing newly created ones), saving e->newPosition before the split and copying it into vm->newPosition afterward. In Step C, we flipped any new edge connecting one old and one new vertex. Finally in Step D, we copied newPosition into position for all vertices.

Sharp corners and edges become progressively smoother with each subdivision iteration, since Loop subdivision computes a weighted average that pulls vertices toward their neighbors. Pre-splitting edges near sharp features adds more geometry in that region, giving the averaging scheme less reach and helping preserve sharpness.

The default cube mesh triangulates each face with a diagonal going in one consistent direction, breaking the cube's symmetry. Since Loop subdivision repositions vertices based on their neighborhood, vertices at corners with asymmetric neighborhoods get pulled unevenly, causing the rounded result to be lopsided. Pre-processing by splitting each face's diagonal and adding cross-edges makes every vertex's neighborhood more symmetric and uniform in valence, resulting in a much more symmetric subdivided shape.
